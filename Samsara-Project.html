Project  Details:

Engine: Unreal Engine 5
Code : C++ and Blueprints
Date Published:  Early Access Late 2024
Development Time: 4 weeks
Project Type: ​Bootcamp / Contract
Company Name:  Infinity 27
Focus: Designing and implementing a spell that fits the game world and design ​parameters

Product Page :   store.epicgames.com/en-US/p/samsara-57a12a/

Spell Pitch

Drop ice shards from the sky, leaving behind an ice floor that has the chance to knock enemies prone. This spell provides powerful crowd control options to remove enemies from combat, affording the player multiple options to proceed.

Gameplay Video

The Design

I came into the project for the bootcamp at the tail end before release, meaning the project had a well defined world set in place. My spell needed to fit the ​theme of the game world following Tibetan Buddhism mythology. Whilst performing research, I was drawn to the idea of ​​the Eight Great Cold Narakas, related to suffering and punishment for negative actions through extreme cold.

I wanted to create a spell that felt powerful that when casted would provide even more gameplay oppertunities to the player. Some of the most fun spell that I had used in other fantasy games and spells like Ice Storm from Baulder's Gate and Blizzard from Final Fantasy came to mind, where I drew my ​inspiration from. 

After play testing the game on an internal build, I found that enemies would group up and swarm the player, so I wanted to create a way to crowd control these groups and turn the favour towards the player.

Initial Design Proposal for the spell

Following Infinity27's production pipeline of spells, I started off with prototyping the proposal to make sure it was do-able.
Using Unreal Engine 5 blueprints I created a concept for the spell to build from.

After sharing the prototype with my mentors and being given the green light, I started work on designing out the spell through UML and Flow charts.

Spell Development ​Process 

The main focuses of the development were spent on creating a system without ​dependencies and easily adjustable for designers​ to tweak how the spell interacted within the game world, through events and editable variables where needed.

To make sure the spell design was robust and working as expected, through each stage of development, I was constantly using debug tools such as the on screen logs, visual gizmos and break points.

For source control the project was using Perforce.

Updating the Visuals

After the spell passed the first set of quality and performance checks, I improved the visuals of the spell. 

Using decals to allow the ice floor to wrap around the terrain. 

To make the spell feel more magical, I added particle effects to the ice shards when hitting the floor

Converting to C++

To improve performance of the spell and keep within the allocated CPU and GPU budgets, the majority of the spells logic was converted from blueprints to C++

My main focuses was to create modular code that was readable so others in the project would be able to review it, understand and update at a later date. I wrapped code into groups using regions and ensured that each function had one responsibility with a clear name for what it does.

An example of this is the code block to the right, showing a function that returns a boolean to decide if we should knock an enemy prone or not. This is called only when needed to save performance. One improvement with this code is to replace the divide operator with a ​multiplication to improve performance, although this may be ​neglectable.

Opening up to Designers

To make sure that designers were able to adjust the spell in an easy way, each variable that needed to be adjustable was opened up to designers in blueprints.

As shown in the code block to the left, when editable variables are used such as the selectable bone name for ragdolls, I make sure to check they are valid before performing functions that would cause errors. 

Performance

To ensure that players will not have a negative experience when casting my spell, suffering frame drops or and lag, I was constantly measuring ​performance statistics  throughout and adjusting where needed.

For example, I first had the try knock enemies prone function called every frame which would cause a lot of stuttering. To fix this, I created a timer that would call the function on intervals whilst the ice floor actor was active.

CPU and GPU testing was done whilst ​casting the spell and within budget, and size map for the spell was in budget at 19.6mb (this is inflated as the blueprint with all its logic still exists as backup for the final presentation).

However, I was only testing in editor so build tests will need to be done too.

Project Thoughts

Overall the bootcamp was a great experience as I got to work in a real team on development of a game that would be published, so I am extremely grateful. On reflection I should have managed my time spent on each stage of development differently. I would spend less time on prototyping and more on the final stages of polishing. 

This experience has me even more excited to join the games industry and I cant wait to see how I will develop as a game designer and programmer!
