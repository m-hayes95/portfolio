Main Focuses

My main focus for this project was to create a team environment that allowed us to work together efficiently and effectively, whilst all enjoying what we were doing during the process of creating the game. Game Design Notes

Before getting started with the project, I first wanted to identify the setting and purpose of the game we were going to make. As a group we decided to create a 3d platformer similar to Mario 64, where there would be a main hub that led to unique worlds. After brainstorming and creating moodboards for the art direction we decided on a fantasy type theme and the main mechanics would be focused around platforming and completing world puzzles with a magical bow and arrow, inspired by Legend of Zelda: BOTW. 

To try to ensure that our team we're passionate about the project, it was important that we all added meaningful contributions to the project. We went over our preferred roles for the project and allocated tasks from the project backlog accordingly. Whilst allocating tasks, we tried to evenly distribute them and those who had more time to commit to the project were given more tasks. 

However, one mistake I made at this time was taking on too much. I was responsible for managing the team and later most of the paperwork, whilst also responsible for level design, and puzzle / world mechanics. For a 13 week period It was too much to take on and we should have cut content.

Creating standards of practice 

As there were multiple programmers on the project, I wanted to make sure that we were following similar practices to avoid any potential problems during mid development. Myself and Robin Pound (see team members) set up a document for our coding and source control practices and then had a team meeting discussing this and answering any questions the team had.

Coding practices:
  • Following principles of SOLID & Agile Scrum
  • Creating clean and readable code 
  • Ensuring the correct use of private and public variables 
  • Using tooltips on editable variables to explain how and what to use them for 
  • Testing features work outside the main scene before adding them 
  • Creating modular code - use of methods and ensuring that scripts had one purpose each
  • Updating trello boards when a task is complete 
Source control practices: 
  • Each feature should have its own branch
  • Test before merge request to the main branch 
  • Peer review of each merge request - cannot be merged by a single person 
  • Do not edit others code or assets without asking permission before hand

Team Work

As not everyone was available everyday, we scheduled as many daily stand ups as possible to discuss the current progress of everyone's tasks, checking if there were any blockers and what was next in the pipeline.
We used Trello to organise tasks between the group members. When some of the group members had less time to commit to the project, we would re-arrange accordingly.

Sound Puzzle Notes

  • Blue Area shows area where UI text displays (for interact)
  • Each cube plays a unique note
  • Lowest cube (refer to image) is the note that shows which order the 3 notes should play in
  • When all 3 notes are hit in the correct order, the puzzle is solved
  • If the wrong notes are played, the puzzle resets
  • Lights on the notes light up when they have been activated, and turn off when the puzzle resets

Checking correct order of notes were played

To validate that the notes were played in the correct order I used the method in the image to the left.

  • First we ​iterate a list of the notes that have just been played through their unique note ID (72)
  • Then we check that order against an array of the note IDs in their correct order (74)
  • If the lists match the order check success variable increases by 1 (83). Once this value reaches 3 (the total number of notes that can be played) the puzzle will be complete 
  • If the lists at any point do not match, the puzzle will be reset (75~80)

L - System

For this project, we wanted to implement some form of L systems. Myself and teammate Robin Pound, worked together to find out how L-systems functioned and created trees which were used within the hub world. However, due to their significant impact on performance, they were limited to a certain size and were not used outside the small hub world.

  • For this L system we generated a random rule (49) combined with a set rule (50)
  • The set rule allowed the plant to maintain some kind of form that we could influence
  • Along with random rotation at the end of branches (within a set range), this allowed us to create a plant that would change everytime it was constructed
  • Link to script - Github

Project Thoughts

Overall, this project was a great learning experience and I am happy with the team and how they performed over the 13 weeks of development. This project also helped me realise the cost of overscopping and the need to allocate more time to the polishing stage of the game. However, everyone on the team wanted to contribute to the coding of the game and having the scope too limited would have caused a lack of work for programmers to do. I definitely took on too much for this project and will need to work letting control go over every aspect of the project.

I also gained a lot of experience with cutting content that was either taking away too many resources, could not be completed to a sufficient standard or was not contributing to the main focus of the game.

Furthermore, I gained more knowledge on how long tasks take to complete, although still not perfect. I have more of an understanding of how long some tasks would take when completing our project backlog and will be able to make more accurate predictions going forward.
